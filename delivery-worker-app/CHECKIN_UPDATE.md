# 打卡流程更新说明

## 更新日期：2024-10-28

## 旧的打卡流程（已废弃）
```
上班打卡 → 装车确认 → 到达现场 → 完工打卡 → 客户签字
（必须按顺序完成，每个订单都要走完整流程）
```

## 新的打卡流程
```
上班打卡（一天只需一次，在仓库）
    ↓
订单1：到达现场 → 完工打卡
    ↓
订单2：到达现场 → 完工打卡（无需返回仓库）
    ↓
订单3：到达现场 → 完工打卡
    ↓
...
```

## 主要变更

### 1. 上班打卡
- **一天只需打卡一次**
- 在第一个订单出发前，在仓库打卡
- 打卡后当天的工作状态变为"已开始"
- 无需每个订单都重复上班打卡

### 2. 取消装车确认
- 移除装车确认打卡环节
- 装车由库管在后台安排（自有车辆/货拉拉等）
- 送货员无需关注装车流程

### 3. 到达现场打卡
- **每个订单都需要打卡**
- 到达客户现场后进行
- 需要定位 + 现场拍照

### 4. 完工打卡
- **每个订单都需要打卡**
- 任务完成后进行
- 需要拍照确认工作成果
- **取消客户签字功能**

### 5. 连续订单处理
- 完成一个订单后，直接前往下一个地点
- 无需返回仓库
- 只需进行到达现场 + 完工两个打卡

## 页面变更

### 打卡页面 (pages/checkin)
- 移除装车确认类型
- 移除客户签字功能
- 简化打卡流程

### 考勤页面 (pages/attendance)
- 只显示上班打卡记录
- 显示每个订单的到达/完工打卡记录
- 更新考勤统计规则

### 工作台首页 (pages/index)
- 添加上班打卡状态提示
- 显示当前进行中的订单
- 快速打卡入口

### 送货单详情 (pages/delivery-detail)
- 时间线只显示：上班 → 到达 → 完工
- 移除装车确认和客户签字
- 根据订单状态显示不同的打卡按钮

## 使用场景示例

### 场景1：普通订单
1. 早上8:00 - 在仓库上班打卡
2. 9:00 - 到达第一个客户现场，到达现场打卡
3. 11:00 - 完成任务，完工打卡
4. 完成！

### 场景2：连续多个订单
1. 早上8:00 - 在仓库上班打卡（只打一次）
2. 9:00 - 订单1：到达现场打卡
3. 11:00 - 订单1：完工打卡
4. 直接前往下一个地点（无需回仓库）
5. 14:00 - 订单2：到达现场打卡
6. 16:00 - 订单2：完工打卡
7. 直接前往下一个地点
8. 17:00 - 订单3：到达现场打卡
9. 19:00 - 订单3：完工打卡
10. 完成！

### 场景3：货拉拉运输
1. 早上8:00 - 在仓库上班打卡
2. 货拉拉将货物送到客户现场
3. 送货员自行前往现场
4. 到达现场打卡
5. 完工打卡
6. 完成！

## 注意事项

1. **上班打卡是必须的**，否则无法进行订单打卡
2. **每个订单必须完成到达现场和完工两个打卡**
3. **装车由库管安排**，送货员无需关心运输方式
4. **取消客户签字**，以完工打卡照片作为完成凭证
5. **连续订单无需返回仓库**，直接前往下一个地点

## 技术实现

### 打卡记录数据结构
```javascript
{
  id: '唯一ID',
  type: 'work' | 'arrive' | 'complete',  // 打卡类型
  date: '2024-10-28',                     // 日期
  time: '09:00',                          // 时间
  orderId: '订单ID',                      // 关联订单（arrive/complete类型）
  orderNo: '订单编号',
  photos: ['照片路径'],                    // 打卡照片
  location: {                             // 定位信息
    latitude: 31.2304,
    longitude: 121.4737,
    address: '地址'
  },
  userId: '用户ID'
}
```

### 检查逻辑
1. 上班打卡检查：一天只能打一次
2. 到达打卡检查：必须先上班打卡
3. 完工打卡检查：必须先到达打卡
4. 订单连续性：无限制，可连续处理多个订单
